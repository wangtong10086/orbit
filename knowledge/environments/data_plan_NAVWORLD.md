# NAVWORLD 数据计划与深度分析

> 最后更新: 2026-03-18 16:15 UTC | 优先级: P0 (GM 杠杆最大) | 状态: v2 训练中

## 1. 环境概述

NAVWORLD 评估 LLM 作为中文旅行规划助手的能力。需要调用高德地图 API 工具（POI 搜索、天气、导航等），最终输出完整的旅行方案。

**评分**: 50 分代码评分（信息一致性 + 完整性 + POI grounding）+ 50 分 LLM 语义评分 = 100 分制。
**格式**: OpenAI function calling（`tool_calls` + `tool` role），Qwen3 原生 `<tool_call>` 格式。
**部署要求**: `tokenizer.apply_chat_template(messages, tools=tools)` + `--tool-call-parser qwen25`。

**战略意义**: 5.7→18 = **+6.4 GM**，全项目 ROI 最高。竞品区间 15.7-25.1，提升空间巨大。

## 3. 瓶颈根因分析

### 3.1 SFT 天花板已确认

| 版本 | 数据量 | 分数 | 变化 |
|------|--------|------|------|
| v10 | 632 条 | 5.1 | — |
| v11 | 2154 条 (+240%) | 5.7 | +12% |

**数据量 ×3.4，分数仅 +12%**。SFT 对 NAVWORLD 已触顶。原因:
1. 数据格式完美但语义质量不足
2. 合成数据有系统性偏差（模板化、qwen3-max 风格固化）
3. 全部 score=1.0，无法区分好/差数据

### 3.2 模板化严重

**核心问题**: 数据过于"完美"和"一致"。
- 仅 3 种工具调用模式
- 方案长度方差极小 (stdev=160 chars)
- 100% 含推理连接词
- 场景高度相似

这导致模型学到的是**一种固定的规划模板**，而非灵活的规划能力。eval 中遇到不同类型的任务（如预算约束、紧急变更、多人不同偏好）时会失败。

### 3.3 POI Grounding 不足

grounding 率 61.8% 意味着每条方案平均漏掉 40% 的工具返回信息。代码评分组件（50 分）直接检查这一点。

**假设**: 如果 grounding 率从 62%→85%，代码评分可能从 ~5→~15。

### 3.4 零错误恢复能力

eval 中工具可能返回空结果（冷门地点、API 限流）。当前数据 0% 错误场景 → 模型遇到空结果时会崩溃。
