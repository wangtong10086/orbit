# SWE-Infinite Environment (replacing SWE-SYNTH)

## Overview

SWE-SYNTH is transitioning to SWE-Infinite: dynamically mined GitHub PR tasks in SWE-bench format. The evaluation format (model behavior) stays the same — what changes is the task source.

## Two Systems

### 1. Task Generation: `affine-swe-infinite` pipeline
- **Location**: `repos/affine-swe-infinite/`
- **Purpose**: Mine GitHub PRs → validate → create SWE-bench format tasks
- **Pipeline**: Auto-discover repos → PR filter → Docker build → patch-split validation → output tasks
- **Output**: JSON tasks with `instance_id, repo, base_commit, patch, test_patch, problem_statement, FAIL_TO_PASS, PASS_TO_PASS, dockerhub_tag`
- **Storage**: R2 `expansion/*.json`, Docker images on `affinefoundation/swe_infinite_images`
- **Languages**: Python (primary), JS/TS, Go, Rust, Ruby
- **Quality**: 3-layer validation, ~60% end-to-end pass rate

### 2. Evaluation: `affinetes` SWE-SYNTH environment
- **Location**: `repos/affinetes/environments/swe_synth/`
- **Task source**: Pre-generated by breaker service (stored in R2), will merge SWE-Infinite tasks
- **Scoring**: Binary 0 or 1 — all required tests (FAIL_TO_PASS + PASS_TO_PASS) must pass

## Model Interaction Format (CRITICAL for training data)

### Response Format
The model responds with **THOUGHT + exactly ONE bash command**:
```
THOUGHT: [reasoning about the bug/next step]

```bash
command_here
```
```

### Multi-turn Flow
1. Model receives: system prompt + problem statement + repo context
2. Model issues one bash command per turn
3. Model sees command output
4. Repeat until model submits fix
5. Submission: `echo COMPLETE_TASK_AND_SUBMIT_FINAL_OUTPUT && git add -A && git diff --cached`

### Key Format Rules
- **NO tool_calls** — all interaction via bash commands in code blocks
- **NO `<think>` tags** — conflicts with THOUGHT format (stripped in eval, but avoid in training)
- **Exactly ONE bash command per response** (can chain with `&&` or `||`)
- **Commands run in subshells** — directory changes don't persist between turns
- **Working directory**: `/app` (repo root)
- Last turn must contain the `COMPLETE_TASK_AND_SUBMIT_FINAL_OUTPUT` marker + git diff

### System Prompt Summary (from affinetes config.yaml)
- "You are a helpful assistant that can interact with a computer shell"
- Response must have THOUGHT section + exactly ONE bash command
- Format violations cause rejection
- Fixer instructions: analyze → reproduce → fix → verify
- Boundaries: modify source code only, NOT tests/config files
- Submission: `git add -A && git diff --cached`

## Breaker Service (Task Pre-processing)
Before eval, a "breaker" service generates bugs:
1. Loads SWE task (gold_patch shows correct code)
2. **BugInjector** agent injects realistic bug (uses LLM)
3. Verifies bug causes test failures
4. **ProblemSummarizer** generates user-facing description
5. Stores to R2 for later evaluation

Bug types: off-by-one, logic-inversion, wrong-variable, missing-check, type-error, wrong-operator, missing-return, exception-swallow

## Training Data Strategy

### What we need
Successful fix trajectories: multi-turn conversations where a model correctly fixes a bug.

### How to generate
1. Run SWE-Infinite pipeline → generate tasks (instance_id, Docker image, problem_statement, etc.)
2. Run strong model (GPT-5.4 / Claude) as fixer agent against tasks
3. Filter: keep only score=1.0 trajectories
4. Format: convert to chat template (system + multi-turn user/assistant)
5. Quality: ensure THOUGHT blocks have clear reasoning, bash commands are targeted

### What good training data looks like
- Model reads relevant files first (`cat`, `grep` to understand codebase)
- Model reproduces the bug (runs failing tests)
- Model identifies root cause in THOUGHT
- Model applies targeted fix (minimal diff)
- Model verifies fix (runs tests again)
- Model submits (COMPLETE_TASK_AND_SUBMIT_FINAL_OUTPUT + git diff)

### Integration with affine-swe-infinite
The `codex_augmenter.py` module already has an agent framework for running models inside Docker containers. This can be adapted for trajectory generation.

- seq < 16384 → most SWE conversations truncated
- Local eval → not possible (needs breaker service + Docker)
- Trailing user messages → model learns wrong prediction target
