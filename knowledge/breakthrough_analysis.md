## 核心约束：几何均值 + 层级权重

排行榜不是简单平均分。算法本质：
- **所有环境子集组合**（L2-L6）都参与计算
- **L6（全环境）权重 = 192**，L2 权重 = 12 → L6 重要性是 L2 的 **16 倍**
- **几何均值**：任一环境为零 → 该子集几何均值崩塌（smoothing 后 0→0.1，但仍是灾难性惩罚）
- **排名衰减**：rank 2 只拿 rank 1 的 50% 权重，rank 3 只拿 25%

**战略含义**：全覆盖 > 单项突破。rank 2 everywhere > rank 1 in half + rank 5 in half。

## 逐环境破局分析

### 1. GAME — 结构性天花板，需方法突破

**现状**: 1,415 条数据，22.6 分，39% non-zero rate

**根因分析**:
- 22 个游戏中，仅 ~9 个可通过 SFT 学习
- goofspiel 已解决（100%）、leduc_poker/bridge/blackjack/euchre 较强（30-100%）
- othello/hex/liars_dice/clobber **永远 0%**（需搜索/博弈论，SFT 无法学会）
- go/chess/checkers/solitaire **不可学习**（永远不投资）
- SFT 天花板约 40-50 分，竞品 45-65 分

**当前 bot 策略胜率**:
| 游戏 | Bot 胜率 | 条数 | 状态 |
|------|----------|------|------|
| gin_rummy | 99% | 297 | bot 数据将 0%→100% |
| goofspiel | 94% | 283 | 已解决 |
| othello | 77% | 232 | 训练后仍 0% |
| liars_dice | 75% | 224 | 训练后仍 0% |
| leduc_poker | 62% | 186 | 接近最优 |
| hex | 62% | 186 | 训练后仍 0% |
| clobber | 54% | 162 | 训练后仍 0% |

**破局路径**:
1. **短期（SFT 优化）**: 增加 hearts/euchre/bridge/blackjack bot 数据（当前缺失），可将 non-zero rate 从 39%→50%+，约 +5-8 分
2. **中期（DPO）**: 589 对偏好对已就绪，beta=0.1，优先用于可学习但不稳定的游戏
3. **长期（RL/MCTS）**: othello/hex 等需要搜索深度的游戏，SFT/DPO 都不够

**数据行动**:
- [ ] 为 hearts、euchre、bridge、blackjack 编写 bot 策略（当前 game_bots.py 只有 7 个游戏）
- [ ] 扩展 gin_rummy bot 多样性（不同策略变体）
- [ ] 目标: GAME 数据从 1,415→2,500+

**预期收益**: 22.6 → 30-35（SFT），35-45（+DPO）

### 2. NAVWORLD — 最大突破口，全员弱势

**现状**: 2,248 条数据，5.7 分，28% non-zero rate

**根因分析**:
- 评分 = 50 分代码评分 + 50 分 LLM 语义评分
- 代码评分 = 信息一致性 × 完整性 的几何均值
- 关键扣分项：
  - 工具多样性 < 3 种 → 惩罚
  - 最终方案 < 800 字 → 完整性扣分
  - POI 名称未在工具返回中出现 → 一致性扣分（反记忆化）
  - 缺少推理连接词（因为/建议/推荐）→ 完整性扣分
  - 交通信息捏造 → 硬性扣分

- **SFT 瓶颈已确认**: +240% 数据仅 +12% 提升
- 竞品 RLStepone 用 RL 方法达到 33.7 分

**数据质量审查**:
- 2,248 条全部有 direction 工具覆盖 ✅
- 100% 使用 OpenAI function calling 格式 ✅
- 100% apply_chat_template 通过 ✅
- 但：**数据全部标记 score=1.0**（自动生成），实际质量参差不齐

**破局路径**:
1. **质量过滤（拒绝采样）**: 用评分器对现有 2,248 条打分，只保留高分数据（可能 1,000→800 高质量）
2. **DPO**: 241 对偏好对，直接在 tool-calling 轨迹上做偏好学习
3. **生成策略升级**:
   - 增加问题复杂度（当前 5 类偏简单）
   - 多轮推理链（先分析需求→工具调用→整合方案）
   - 错误恢复场景（工具返回空→重试不同参数）
4. **格式对齐强化**: 确保 sglang 部署用 `--tool-call-parser qwen25`

**数据行动**:
- [ ] 对现有数据用评分逻辑打分（forge/data/sft.py 中 validate_navworld）
- [ ] 移除低质量条目（最终方案<800字、工具<3种、POI 不grounded）
- [ ] 新增复杂场景：多日行程、跨城组合、预算约束
- [ ] 目标: 1,500 条高质量 > 2,248 条混合质量

**预期收益**: 5.7 → 15-20（质量过滤+数据优化），20-30（+DPO）

### 3. SWE-SYNTH — 序列长度是唯一瓶颈

**现状**: 1,351 条数据，~31 分

**根因分析**:
- **98% 数据在 seq=4096 时被截断**（中位长度 ~16K chars）
- 模型只学到对话开头（系统提示+问题描述），从未见过完整修复过程
- 二元评分（0 或 1），没有部分分
- 无法本地评估（需 breaker service）
- 不支持 think tags（与 THOUGHT 前缀冲突）

**数据质量审查**:
- 1,351 条全部 score >= 0.5 ✅
- 全部 <= 32K chars ✅
- 全部最后消息为 assistant ✅
- 但在 seq=4096 训练时，绝大部分截断为无效数据

**破局路径**:
1. **seq=8192 训练**（v12 方案，从未完成）: 将可用数据比例从 ~2%→~30%
2. **数据筛选**: 只保留 <= 8192 tokens 的完整对话（约 200-400 条）
3. **DPO**: 258 对偏好对可用

**数据行动**:
- [ ] 按 token 数重新统计: 多少条在 4096/8192/16384 tokens 内完整
- [ ] 为 seq=8192 训练准备数据子集
- [ ] 考虑对话压缩: 移除冗余的中间输出，保留关键修复步骤

**预期收益**: 31 → 35-40（seq=8192），但无法验证直到部署

### 4. LIVEWEB — 当前无解，需上游改造

**现状**: 430 条数据，~24 分

**根因分析**:
- **数据长度灾难**: 中位 145K chars（~36K tokens），99.5% > 16K chars
- 即使 seq=8192，仍然 0% 数据可完整容纳
- DashScope 合成尝试全部失败（13/13 qwen3-max, 3/3 coder-plus, 5/5 3.5-plus）
- 合成失败原因: LLM 无法生成正确的 JSON action + 无法真实导航页面
- 评分器: 全员 16-28 分，差异极小

**上游压缩方案**（需修改 liveweb-arena 源码）:
1. **DOM 压缩**: 移除非交互装饰元素 → 预期 65% 压缩
2. **页面去重**: URL+title 不变时只发 delta → 50-70% 去重
3. **标准化工具调用**: 自定义 JSON → OpenAI function calling
4. **历史步骤压缩**: 旧步骤只保留 action_type+result

**预期效果**: 39K tokens → 8-10K tokens/条，可训练比例 18%→70%+

**数据行动**:
- [ ] **不投入更多数据工作**（ROI 极低）
- [ ] 保持 430 条作为 "非零安全网"
- [ ] 如果用户授权，可尝试修改 liveweb-arena 上游做压缩

**预期收益**: 维持 ~24 分（无上游改造则无法提升）

## 综合破局策略

### 优先级排序（按几何均值影响）

| 优先级 | 行动 | 环境 | 预期排名提升 | 成本 |
|--------|------|------|--------------|------|
| P0 | NAVWORLD 数据质量过滤+拒绝采样 | NAVWORLD | 5.7→15+ | 低（已有数据） |
| P0 | seq=8192 训练 | SWE-SYNTH | 31→35+ | 中（$9/run） |
| P1 | 新增 hearts/euchre/bridge/blackjack bot | GAME | 22.6→30+ | 低（编码工作） |
| P1 | DPO on NAVWORLD (241 pairs) | NAVWORLD | 15→20+ | 中（$9/run） |
| P2 | DPO on GAME (589 pairs) | GAME | 30→40+ | 中（$9/run） |
| P2 | 数据 mix 重新平衡 | 全局 | GM +3-5 | 低 |
| P3 | LIVEWEB 上游压缩 | LIVEWEB | 24→30+ | 高（改上游） |

### 目标场景

**保守估计**（仅 SFT 优化）:
- GAME 30, NAVWORLD 15, SWE-SYNTH 35, LIVEWEB 24, LGC-v2 95, PRINT 80
- GM ≈ 37.8（不够 #1）

**中等估计**（SFT + DPO）:
- GAME 40, NAVWORLD 25, SWE-SYNTH 38, LIVEWEB 24, LGC-v2 95, PRINT 82
- GM ≈ 44.3（接近 #1 的 41.4）

**乐观估计**（SFT + DPO + 上游压缩）:
- GAME 45, NAVWORLD 30, SWE-SYNTH 40, LIVEWEB 30, LGC-v2 95, PRINT 85
- GM ≈ 50.2（超越 #1）

### 关键认知

1. **LIVEWEB 是几何均值的锚点** — 24 分将全局 GM 压低 ~15%。没有上游压缩，GM 上限约 45
2. **NAVWORLD 是最高杠杆** — 全员弱势，5.7→25 的提升对 GM 贡献最大
3. **SFT 到 DPO 的转折点已到** — NAVWORLD 和 GAME 都确认了 SFT 瓶颈
4. **数据质量 > 数量** — 2,248 条混合质量 < 1,000 条高质量（NAVWORLD 场景）
5. **seq=8192 是 SWE-SYNTH 的免费午餐** — 唯一改变是训练参数，不需要新数据
